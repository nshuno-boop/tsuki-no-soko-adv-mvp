# Phase 5.5 Playtest QA

## 概要

Phase 5 Alpha Polish後の実機プレイテスト前QAとして、ルート、文章テンポ、推理UX、UI視認性、技術安定性の5観点で静的監査を行った。現時点でP0相当の進行不能・起動不能リスクは見つかっていない。

ただし、Ren'Py標準の終了確認画面が不足していることにより、ウィンドウ終了時に `Layout object has no attribute 'yesno_prompt'` が発生する実行時リスクを確認した。また、最終章で「月生まれ医療評価」が不足した場合に、白石アカリの追加聞き込みへ戻る導線がやや弱い。

## A. Route QA

### 危険箇所

- True Endingは正解数だけでなく、重要失敗0、Q6/Q8/Q9正解、最終必須証拠完備が必要。
- 第3章から第4章へは4人の初回聞き込みで進めるが、アカリを深掘りしないまま進むと「月生まれ医療評価」が不足しやすい。
- 証拠不足のまま進む選択はBad Endingへ直行するため、警告文で意図を明確にする必要がある。

### 実機で重点確認すべき箇所

- 第3章で0〜3人聞き込み時に不足人数と未聞き込み対象が表示されるか。
- 第3章で4人以上聞き込み後、徹の監査ファイルが入手され、第4章へ進めるか。
- アカリ未深掘りのまま第5章へ進んだ場合、不足警告から戻って医療評価を取れるか。
- Q6/Q8/Q9の正否でTrue / Normal / Badの境界が意図どおり変わるか。
- 証拠不足のまま進むと、最終推理に入らずBad Endingへ到達するか。

### 軽微修正候補

- 第5章のおすすめ行動に、医療評価不足時のアカリ再聞き込み案内を追加する。
- 証拠不足警告に、Bad Endingへ分岐する旨を明記する。
- PLAYTEST_CHECKLISTに分岐境界とアカリ再聞き込みの確認項目を追加する。

## B. Text Pacing

### 削れる文

- 最終章ハブの説明文はやや説明的なため、「手口だけでは足りない」程度に短縮する。
- 最終推理開始前の構造説明は答えの骨格を言いすぎるため、短くする。

### 残すべき文

- 徹の最後の一文はテーマを支えているため残す。
- True EndingのALMAとノアの記録会話は感情の核として残す。
- Bad Endingの公式記録に残らない締めは、冷たい結末として有効。

### 余韻が足りない箇所

- True Endingは「記録しました。」の直後に制度説明へ進むため、1拍の情景描写がほしい。
- Normal Endingはノアの感情から政治的結果へ急に移るため、澪の沈黙を1文足すと自然。
- Bad Endingはノアの問いのあと、誰も答えない間を1文置くと成立感が増す。

### 軽微修正候補

- 最終章導入を短縮する。
- True / Normal / Bad Endingに各1文程度の余韻を足す。

## C. Deduction UX

### 分かりやすい点

- Q1〜Q9は、死因、ALMA誤認、白兎3号、人為干渉、仕様悪用、アリバイ、実行者、動機、徹の最後の行動という順で自然に積み上がる。
- 複数証拠問題は必要証拠数を表示しており、単一証拠問題との差が分かる。
- True必須のQ6/Q8/Q9は、実行可能性、動機の深さ、物語的真相に対応しており納得感がある。

### 誤解されそうな点

- Q2は単一選択だが成功文が複数証拠提示のように読める。
- Q6のヒントは答えに近いため、やや推理感を削る。
- Q7の成功文はQ8前に動機まで言い切っている。
- Q9のヒントは正解証拠へ直行しやすい。

### 文言の修正候補

- Q2の問題文・成功文を「判断材料」寄りに調整する。
- Q6/Q9のヒントを証拠名直結から一段抽象化する。
- Q7の成功文を実行者特定に絞る。
- Q8の成功文に、動機は説明であって免責ではないことを加える。
- 複数証拠UIの選択数を「選択中 / 必要数」で表示する。

## D. UI Readability

### はみ出しリスク

- 1280x720では主要画面の縦余白が少なく、長文追加時に下端が詰まりやすい。
- 証拠詳細は固定枠で、今後さらに説明が伸びると縦方向の余裕が弱い。
- タイムラインは1行固定文言が多く、文言追加時に横方向の余裕が減る。

### 視認性リスク

- おすすめ行動は有効だが、ヘッダー内の小さい黄色文字で埋もれる可能性がある。
- 重要証拠表示は一部ボタン名のみ「重要:」付きで、不統一に見える。
- 補助テキストのサイズ14〜15は、720pでやや弱く見える可能性がある。

### 軽微修正候補

- 重要証拠表示を全件で統一する。
- 証拠詳細・人物メモ・タイムラインにスクロール余地を持たせる。
- おすすめ行動を目的表示と視覚的に分ける。
- 今回は大きなUI改修を避け、実機で読みにくい箇所を確認項目として残す。

## E. Technical Stability

### 技術的な懸念

- 終了時の確認画面が不足しており、quit操作で `yesno_prompt` 例外が出る。
- 複数証拠推理は `Return(selected_ids)` がsetで返る前提のため、想定外の戻り値はlintで拾いにくい。
- debug_menuは `config.developer` ガード済みだが、ジャンプ後の状態差分は実機確認が必要。

### lintで拾えなそうなリスク

- GitHub ActionsのStatic checkはRen'Py本体のlint / compileを含まない。
- `show screen` 参照、標準確認画面、screen戻り値型は既存の整合性チェックだけでは拾いきれない。
- 720pでの実際の文字密度やモーダル重なりは静的検査では判断しきれない。

### 軽微修正候補

- `confirm` screenを追加して終了確認を復旧する。
- `check_project_integrity.py` に `show screen` と標準確認画面の存在確認を追加する。
- CIへのRen'Py lint追加は環境依存があるため、今回は後回しにする。

## 実機プレイ時に確認すべき項目

- True Ending: 必須証拠完備、Q6/Q8/Q9正解、重要失敗0で到達できるか。
- Normal Ending: Q6/Q8/Q9の一部を外した場合、Trueにならず苦い解決として成立するか。
- Bad Ending: 証拠不足のまま進むと、ALMA誤作動説に流れる結末として成立するか。
- 医療評価導線: アカリ未深掘りから不足警告へ進み、戻ってアカリ再聞き込みで復帰できるか。
- UI: 証拠品一覧、人物メモ、タイムライン、複数証拠UI、目的表示が1280x720で読めるか。
- 終了操作: ウィンドウを閉じてもtracebackが出ないか。
